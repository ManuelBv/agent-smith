# Multi-Agent LLM Systems: Architecture, Cost, and Quality Research

Research date: 2026-07-08. Compiled for Manuel's `apps/agent-smith` example projects, building on the patterns already implemented in `apps/ceiling-designer` (strict sequential pipeline with file-based handoff) and `apps/purrfect-blocks` (orchestrator + parallel subagents + parallel verification).

---

## 1. Executive Summary

**The core finding, stated plainly: decomposition into agents does not reduce hallucination by itself. What reduces hallucination is *asymmetric verification* — a separate check that is structurally easier than the original generation task — combined with *tool-grounding* (forcing claims through external, checkable sources) and *bounded iteration* against a concrete pass/fail gate.** Multi-agent-ness (more agents, more parallelism, more roles) is a orthogonal, mostly cost/latency lever, not a quality lever. Several rigorous 2025-2026 papers now show single agents matching or beating multi-agent setups on non-parallelizable tasks at a fraction of the cost ([Tran & Kiela 2026](https://arxiv.org/abs/2604.02460); [Cemri et al., MAST 2025](https://arxiv.org/html/2503.13657v3); a 64%-of-tasks single-agent-wins result reported in [Digital Applied's 2026 survey](https://www.digitalapplied.com/blog/multi-agent-orchestration-5-patterns-that-work)).

Manuel's two existing systems already have the two things the evidence says actually matter:
1. **Independent verification that doesn't trust the generator's self-report** — ceiling-designer's Reviewer re-verifies from scratch; purrfect-blocks' code-reviewer + unit-tester run independently of the implementer.
2. **Bounded retry loops with an explicit gate** — max 5 rounds in ceiling-designer, structured JSON severity + approval gate in purrfect-blocks.

What neither system yet does, and what the evidence supports adding cheaply:
- **Use a *different, stronger* model for the judge/verifier role than for the generator role.** This is the single highest-leverage, cheapest change available — it directly exploits generator-verifier asymmetry (§3) and avoids same-model self-preference bias (§3, §4).
- **Route the verifier through tool-grounded checks (tests, linters, type-checkers, accessibility audits) wherever a checkable ground truth exists**, before falling back to LLM judgment for the parts that are genuinely subjective (naming, prose quality, spec fidelity).
- **Recognize where the pipeline is currently more sequential than it needs to be** — ceiling-designer's Researcher step and purrfect-blocks' parallel research agents show fan-out already works; the next unlock is running independent *reviewers* in parallel and reconciling disagreement (a light debate/voting step) rather than trusting a single reviewer pass.

**On the $100-one-shot vs. $8-10-multi-agent question specifically:** the honest answer is that multi-agent decomposition is *not generally cheaper* — Anthropic's own research system uses **~15x the tokens** of a single chat call ([Anthropic, 2025](https://www.anthropic.com/engineering/multi-agent-research-system)), and other data shows single-agent runs using *fewer* tokens than multi-agent runs in some settings ([BAMAS, 2025](https://arxiv.org/pdf/2511.21572)). The $8-10 number is achievable, but only via **model tiering** (cheap/fast model for drafting and research fan-out, expensive model reserved for the judge/reviewer/planner steps) and **avoiding redundant context re-derivation** across agents — not from "more agents = less total work." See §4 for concrete math.

## 2. Core Architectural Patterns

### 2.1 Sequential / Pipeline
One agent's output is the next agent's input, strictly ordered, no re-entry except explicit backward loops. **Wins when**: steps have a genuine, unavoidable data dependency (you cannot plan before you've spec'd; you cannot implement before you've planned) and step quality is easier to check than to parallelize. **Fails when**: forced serialization of independent work wastes wall-clock time, and when an early-stage error propagates uncaught through every downstream stage ("error compounding," explicitly named as a top risk in [Anthropic's writeup](https://www.anthropic.com/engineering/multi-agent-research-system): "minor system failures can be catastrophic for agents"). **Maps to**: `ceiling-designer` end-to-end — Speccer → Researcher → Planner → Implementer → Reviewer is a textbook pipeline, with the loop-back from Reviewer to Implementer being the one place non-linearity is deliberately reintroduced, bounded to 5 rounds.

### 2.2 Parallel / Fan-out-Fan-in
Multiple agents work simultaneously on independent subtasks or independent samples of the same task; a final step aggregates (by voting, merging, or LLM synthesis). **Wins when**: subtasks are genuinely independent (no shared mutable state), the aggregation step is cheap and reliable, and latency matters — Anthropic reports parallel tool-calling and subagent fan-out **cut research time by up to 90%** for complex queries. **Fails when**: subtasks are only superficially independent — [PlanCraft benchmark results](https://arxiv.org/abs/2604.02460) show multi-agent coordination on sequential tasks caused up to **70% performance degradation** because agents lacked visibility into each other's evolving state. **Maps to**: purrfect-blocks' `architecture-researcher` + `legal-researcher` running in parallel, and `code-reviewer` + `unit-tester` running in parallel post-implementation — both are correctly-identified independent subtasks.

### 2.3 Orchestrator-Supervisor
A lead/orchestrator agent (often a stronger, more expensive model) plans, decomposes, dispatches to specialized workers, and synthesizes. This is Anthropic's own production architecture for Claude's Research feature: Opus 4 as lead, Sonnet 4 as workers, **90.2% improvement over single-agent Opus 4** on their internal research eval ([Anthropic, 2025](https://www.anthropic.com/engineering/multi-agent-research-system)). **Wins when**: the task decomposes cleanly into parallelizable subtasks, and the value of the answer justifies materially higher token spend (15x a normal chat call, per Anthropic's own numbers). **Fails when**: the orchestrator itself is under-specified — Anthropic explicitly logged early failures like agents "spawning 50 subagents for simple queries" and continuing "when they already had sufficient results," and the current architecture still runs subagents **synchronously** (lead waits for a full round before proceeding), which is itself a bottleneck they flag as unresolved. **Maps to**: purrfect-blocks' opus-tier orchestrator dispatching to specialized subagents is precisely this pattern, already correctly gated with parallel dispatch only where independence is real.

### 2.4 Debate
Two or more agent instances (often same model, different seeds/personas, or genuinely different models) argue for/against a position over multiple rounds, converging on a shared answer or handing off to a judge. [Du et al. 2023](https://arxiv.org/abs/2305.14325), the foundational debate paper, showed multi-round debate improves mathematical/strategic reasoning and factual accuracy versus single-shot generation on the same base model. But more recent work is more cautious: [Stay Focused: Problem Drift in Multi-Agent Debate (2025)](https://arxiv.org/pdf/2502.19559) documents debate rounds drifting away from the original problem statement as rounds accumulate, and [The Confident Liar (2026)](https://arxiv.org/pdf/2606.10296) shows debate participants can converge confidently on a *wrong* shared answer when neither has grounding to correct the other — debate improves consistency, not necessarily correctness, unless grounded. **Wins when**: the task is adversarial/argumentative by nature (red-team vs. blue-team, "does this spec meet the acceptance criteria" as prosecution/defense) or when paired with an independent judge and a hard round cap. **Fails when**: there's no external ground truth to break ties, or rounds run unbounded (drift). **Maps to**: neither of Manuel's systems currently implements debate — this is a genuine gap. Ceiling-designer's Reviewer is closer to a *verifier* than a *debater* (good), but could be strengthened by having two independent reviewer instances vote before invoking the retry loop, rather than trusting a single reviewer pass.

### 2.5 Hierarchical / Nested Subagents
Orchestrators that themselves dispatch to sub-orchestrators, recursively — e.g., a "backend lead" subagent that further spawns a "migration-writer" and "API-writer." **Wins when**: the problem has natural multi-level decomposition (a monorepo with distinct domains) — context-isolation studies show a single agent loading rules across a whole monorepo pulls ~16,800 tokens of rules before touching code, vs. ~3-4,300 tokens for a domain-scoped subagent, a **4x reduction in irrelevant context** ([claudefa.st sub-agent best practices](https://claudefa.st/blog/guide/agents/sub-agent-best-practices)). **Fails when**: nesting depth exceeds what a human can debug — failures become hard to trace (this is one of the three MAST failure categories, §4). **Maps to**: not present in either of Manuel's systems today; not needed at his current project scale (single-feature web apps), and the evidence suggests it should only be added when a genuine multi-domain split emerges (e.g., separate frontend/backend/infra concerns within one feature).

### 2.6 Swarm / Decentralized (peer-to-peer, groupchat)
No fixed lead; agents communicate directly (AutoGen's GroupChat, CrewAI's crew-of-peers, OpenAI's original Swarm handoff-as-tool-call model). [OpenAI Swarm](https://github.com/openai/swarm) modeled this as "a handoff is just a tool call that returns another Agent," sharing one conversation thread; it's since been retired in favor of the production-grade [OpenAI Agents SDK](https://openai.github.io/openai-agents-python/multi_agent/), which frames the choice explicitly as **agents-as-tools** (bounded subtask, no conversation takeover) vs. **handoffs** (full ownership transfer). **Wins when**: routing itself is dynamic and not knowable in advance (a support bot that hands off to billing vs. tech-support vs. escalation). **Fails when**: task structure is actually known upfront — swarm/groupchat models pay a real coordination tax for flexibility you don't need. Microsoft has since moved AutoGen itself to maintenance mode in favor of a more structured Agent Framework ([pecollective 2026 comparison](https://pecollective.com/blog/ai-agent-frameworks-compared/)), a signal that the industry is converging away from unstructured groupchat toward more supervised/graph patterns for production use. **Maps to**: not used by Manuel, and correctly so — his pipelines have fully-known-in-advance structure (spec → research → plan → implement → review), which is exactly the case where a fixed pipeline/orchestrator beats a decentralized swarm.

## 3. What Actually Reduces Hallucination / Spec Drift (Ranked by Evidence Strength)

1. **Tool-grounding / external verifiability (strongest evidence).** Anything checkable by a deterministic tool (tests passing, types checking, a citation actually existing in the source doc, an accessibility rule actually firing) removes the LLM from the judgment loop entirely for that check. Anthropic's CitationAgent pattern — a dedicated pass that verifies every claim against actual source locations *after* the research is drafted — is exactly this: separate the "can this be checked mechanically" work from "is this a good synthesis" work ([Anthropic, 2025](https://www.anthropic.com/engineering/multi-agent-research-system)). This is why ceiling-designer's TDD red-green-refactor loop is doing real work: tests passing is a ground truth no LLM self-assessment can substitute for.

2. **Generator-verifier asymmetry, exploited by using a stronger/different model as verifier.** The theoretical grounding: verifying a solution is frequently far cheaper than producing one — true in optimization (dual certificates), factorization, maze-solving, and reasoning traces alike ([Understanding Verification Dynamics in LLMs, 2025](https://arxiv.org/html/2509.17995v2)). Concretely: a model scoring only ~15% accuracy *generating* proofs scored **85% accuracy** when merely asked to *judge* proof correctness ([tobysimonds.com, 2025](https://tobysimonds.com/research/2025/09/29/Proofs.html)). This is the single strongest, most directly actionable finding for Manuel: **the verifier step doesn't need the generator's full capability — it needs a different vantage point.** In practice this means both (a) a structurally distinct verification task (checklist/rubric grading rather than "redo the work and compare"), and (b) ideally a different/stronger model instance so it isn't reproducing the generator's own blind spots.

3. **Same-model / self-preference bias is real and works against naive self-review.** LLM judges systematically favor outputs from themselves or their own model family — GPT-4 as judge showed self-preference bias ranging from **-38% to +90%** across benchmarks depending on setup ([arxiv 2410.21819](https://arxiv.org/html/2410.21819v1); confirmed and quantified further in [arxiv 2604.22891](https://arxiv.org/pdf/2604.22891)). This is direct evidence for something Manuel's ceiling-designer design already anticipated ("Reviewer does NOT trust implementer's self-assessment") — but the research suggests going one step further: **the reviewer itself should ideally not be the same model/prompt-family as the implementer**, or at minimum should use a structurally different evaluation method (rubric-based, cognitive-load-decomposed scoring reduces self-preference bias by ~31.5% per [arxiv 2604.06996](https://arxiv.org/abs/2604.06996)) rather than free-form "does this look right to me."

4. **Multi-agent debate — real but narrower and driftier than the 2023 paper suggested.** [Du et al.'s original result](https://arxiv.org/abs/2305.14325) (multi-round debate improves factuality/reasoning) holds up for math/logic/strategic-reasoning benchmarks. But newer work shows two failure modes that weaken it as a general hallucination-fix: **problem drift** across debate rounds ([arxiv 2502.19559](https://arxiv.org/pdf/2502.19559)) and **confident convergence on a shared wrong answer** when no participant has grounding to correct the other ([arxiv 2606.10296](https://arxiv.org/pdf/2606.10296)). Verdict: debate is useful as a *disagreement-surfacing* mechanism (if two independent reviewers give different verdicts, that's a real signal something's ambiguous) but should not be trusted as a truth-generating mechanism on its own — it needs to bottom out in a tool-grounded or human check.

5. **Self-consistency / majority voting over N samples — solid for narrow, checkable-answer tasks; weaker for open-ended synthesis.** Sampling N reasoning paths from one model and majority-voting the final answer is well-established for math/logic-style problems with a single correct answer, and cheap process/outcome reward models (some as small as **0.6B parameters** per [TrajSelector, 2025](https://arxiv.org/pdf/2510.16449)) can rerank Best-of-N candidates efficiently. This doesn't transfer cleanly to open-ended tasks like "write a good spec" or "review this component's accessibility" — there's no discrete answer to vote over. Best used inside narrowly-scoped subtasks (does this regex match all test cases; does this function have an off-by-one) not at the level of "is this whole feature correct."

6. **Reflexion / Self-Refine (single-agent iterative self-critique) — the weakest lever of the group for hallucination specifically, useful for other things.** [Reflexion](https://arxiv.org/html/2303.11366) shows strong gains (+22% absolute on AlfWorld) when there's an environment-provided feedback signal (task succeeded/failed) to reflect on — it is not a hallucination-fix, it's a learning-from-failure-signal mechanism. Without an external feedback signal, self-refine degrades to the same self-preference-bias problem as self-review (item 3). Useful in Manuel's systems specifically where there *is* an external signal (test failures, lint errors) — much less useful as a substitute for independent review of subjective spec-fidelity questions.

**Ranking summary for Manuel's actual decision:** invest first in (1) tool-grounding everywhere a ground truth exists, then (2) a genuinely different/stronger verifier model, then (3) structurally distinct rubric-based review prompts to blunt self-preference bias. Debate and voting are nice-to-haves for specific narrow sub-questions, not the main lever.

## 4. Cost/Quality Tradeoff — the "$100 one-shot vs $8-10 multi-agent" question

### The uncomfortable baseline finding
Anthropic's own data point: their multi-agent research system uses **~15x more tokens than a single chat turn**, and a single agent uses ~4x a chat turn — meaning multi-agent is roughly **~3.75x more expensive in raw tokens than an equivalent single-agent run**, before counting orchestration overhead ([Anthropic, 2025](https://www.anthropic.com/engineering/multi-agent-research-system)). Separately, **token usage alone explained 80% of the performance variance** on their BrowseComp eval — i.e., much of the quality win from "multi-agent" was actually just a proxy for spending more total compute, not a structural advantage of the agent topology itself. This is corroborated by [Tran & Kiela (2026)](https://arxiv.org/abs/2604.02460): under an *equalized* thinking-token budget, single-agent LLMs matched or beat multi-agent setups on multi-hop reasoning, concluding "many reported advantages of multi-agent systems are better explained by unaccounted computation and context effects rather than inherent architectural benefits."

So: **decomposing into agents is not inherently a cost-saving move.** The $8-10 target is achievable, but only through three specific, separable levers — and it's important for Manuel to know which lever is doing the work:

**Lever 1 — Model tiering (the actual source of savings).** Route cheap/fast models (Haiku-tier) to drafting, research fan-out, and mechanical tasks; reserve expensive models (Opus-tier) only for planning and final judgment. Anthropic's own system already does this (Opus lead, Sonnet workers). Napkin math at illustrative per-token pricing ratios (cheap:mid:expensive roughly 1:5:15, consistent with published Anthropic/OpenAI tiered pricing as surveyed in [pecollective's 2026 LLM pricing comparison](https://pecollective.com/blog/llm-api-pricing-comparison/)): if a single frontier-model one-shot run costs ~$100 in tokens at the expensive tier, and you can push 70-80% of that token volume (research, drafting, boilerplate implementation) down to the cheap tier while keeping only the ~20-30% highest-leverage tokens (planning, final review/judge) at the expensive tier, you land near $25-35 just from tiering — real, but not yet $8-10.

**Lever 2 — Context scoping (avoiding full-context re-derivation per agent).** The other real saving is each subagent loading only the ~3-4k tokens of context relevant to its narrow task instead of the full ~15-20k token project context every step — the 4x reduction in irrelevant context cited by [claudefa.st](https://claudefa.st/blog/guide/agents/sub-agent-best-practices). This compounds with tiering: cheap model + small context per call is where multi-agent decomposition can genuinely beat "always send the frontier model the entire growing context." Combined with tiering, this is realistically how you get toward the $8-10 range — but it requires deliberately keeping each agent's prompt small and file-based (exactly what ceiling-designer's `N-agent-output.md` handoff pattern already does — it hands the *next* agent a distilled artifact, not the full prior conversation).

**Lever 3 — where multi-agent COSTS MORE, not less (the actual risk to watch for):**
- **Redundant context re-derivation**: if each subagent has to re-read the whole codebase/spec from scratch because the handoff artifact is incomplete, you pay full context cost *N times* instead of once. This is the difference between a *good* file-based handoff (ceiling-designer's approach) and a naive one.
- **Excessive back-and-forth / unbounded retries**: without a hard round cap, a reviewer-implementer loop can run indefinitely, multiplying cost with no guaranteed convergence — this is precisely why ceiling-designer's 5-round cap and purrfect-blocks' structured JSON gate matter; they're cost controls as much as quality controls.
- **Coordination/orchestration overhead itself**: the orchestrator's own planning and synthesis calls are pure overhead relative to a single-agent baseline — worth it only when parallelization genuinely saves wall-clock time or the fan-out genuinely covers ground a single context window couldn't (Anthropic's stated justification for when multi-agent is worth it: "tasks that involve heavy parallelization, information that exceeds single context windows, and interfacing with numerous complex tools").
- **The synchronous bottleneck**: Anthropic's own lead agent still waits synchronously for each full round of subagents before proceeding — meaning naive multi-agent pipelines can be simultaneously *more expensive and not faster*, the worst of both worlds, unless async dispatch is deliberately engineered.

**Bottom line math for Manuel:** the path to $8-10 isn't "add more agents," it's "keep the same 5-stage pipeline shape he already has, tier the models within it (cheap for Speccer interview/Researcher drafting, mid-tier for Implementer, expensive only for Planner's human-gated decision and Reviewer's final judgment), and keep the file-based handoffs lean." That's a more surgical, cheaper path than either a giant single frontier call or an undisciplined swarm of agents.

## 5. Known Failure Modes and Mitigations

Grounded primarily in [Cemri et al., "Why Do Multi-Agent LLM Systems Fail?" (MAST), 2025](https://arxiv.org/html/2503.13657v3) — analysis of 1,600+ annotated execution traces across 7 frameworks, 14 distinct failure modes, inter-annotator agreement κ=0.88 (a rare rigor level for this literature; treat as the most trustworthy taxonomy currently available).

| Failure category | Share of failures (MAST) | Concrete manifestation | Mitigation |
|---|---|---|---|
| **Specification & system design flaws** | ~41.8% | Ambiguous role definitions, poor task decomposition, duplicate agent roles, missing termination conditions | Explicit task boundaries per agent (Anthropic: "an objective, an output format, guidance on tools/sources, clear task boundaries"); explicit stop conditions, not implicit "keep going until it feels done" |
| **Inter-agent misalignment** | ~36.9% | Communication breakdowns, **context loss during handoffs**, conflicting outputs, format mismatches | File-based structured handoffs with a fixed schema (exactly ceiling-designer's `N-agent-output.md` pattern) rather than passing raw conversation history |
| **Verification / termination gaps** | remainder | Agents stopping too early or never stopping; no one checks the final state against the original goal | A dedicated, independently-scoped review stage that re-derives correctness from the *original spec*, not from the prior agent's claims |
| **Error compounding / amplification** | — | Uncoordinated multi-agent systems amplify errors up to **17x**; centralized architectures with a validation bottleneck contain amplification to ~**4.4x** (MAST) | Centralize verification at one gate rather than trusting every intermediate step; this is precisely what a single strong Reviewer stage (vs. distributed peer trust) buys you |
| **Same-model / self-preference bias** | — (separate literature, §3) | Reviewer built from same model/prompt family over-trusts implementer's own class of mistakes | Different model tier/family for verifier where feasible; rubric-based (not free-form) review prompts |
| **Cost explosion from redundant re-derivation** | — | Each subagent re-reading full context because handoff artifacts are incomplete | Distilled, purpose-built handoff artifacts; Anthropic's own "artifact pattern" — let specialized agents write persistent outputs directly rather than routing everything back through a lead agent that re-copies it |
| **Orchestrator-induced errors** | folds into spec/design flaws | Orchestrator spawns too many/too few subagents, misjudges task complexity (Anthropic's real incident: 50 subagents spawned for a simple query) | Give the orchestrator explicit effort-scaling heuristics ("simple fact-finding: 1 agent, 3-10 tool calls; comparisons: 2-4 subagents, 10-15 calls each") rather than leaving scaling to its judgment alone |

Additional corroborating point: sequential-task coordination can produce up to **70% performance degradation** relative to single-agent execution when agents are forced to coordinate on tasks that don't actually parallelize ([PlanCraft results reported via Tran & Kiela, 2026](https://arxiv.org/abs/2604.02460)) — a strong argument for keeping genuinely sequential work (like ceiling-designer's pipeline) sequential rather than artificially parallelizing it.

## 6. Concrete Recommendations for Building Example Projects in Claude Code

1. **Keep using file-based handoffs as the default memory model** — this is already validated by both the MAST findings (context loss at handoffs is ~37% of failures) and Anthropic's own artifact pattern recommendation. The fix for handoff failure isn't "pass more context," it's "pass a smaller, well-specified artifact." Treat each `N-agent-output.md` as a contract/schema, not a transcript.

2. **Use Claude Code's Task/Agent tool for parallel fan-out only where subtasks are truly independent** — architecture-researcher + legal-researcher in parallel is correct because neither's output changes the other's inputs; code-reviewer + unit-tester in parallel post-implementation is correct for the same reason. Do not parallelize Speccer→Researcher→Planner-style dependency chains — the evidence (PlanCraft, 70% degradation) says this actively hurts.

3. **Tier models by role, not by agent identity.** Concretely for a 5-stage pipeline like ceiling-designer: Speccer and Researcher (drafting/gathering — no irreversible decisions) can run on a cheaper/faster model; Planner (human-gated, irreversible decision) and Reviewer (final quality gate) should run on the strongest available model. This is the single highest-leverage lever for hitting an $8-10 cost target without sacrificing the quality of the two steps that matter most (the ones a mistake in is expensive to unwind).

4. **Make the Reviewer/verifier structurally distinct from the Implementer, not just instructed to be skeptical.** Two concrete options, both cheap to add: (a) use a different model for review than for implementation where cost allows; (b) even on the same model, switch the review prompt from "check if this is good" (free-form, prone to self-preference bias) to an explicit rubric/checklist scored dimension-by-dimension (cuts self-preference bias ~31.5% per [arxiv 2604.06996](https://arxiv.org/abs/2604.06996)) — this maps directly onto ceiling-designer's existing Farley's-8-gates self-check, which should ideally be *re-run independently by the Reviewer*, not inherited from the Implementer's self-report.

5. **Keep bounded retry loops with hard caps and a visible log** — both existing systems already do this (5-round cap + Review Runs Log; structured JSON approval gate). This is directly validated: unbounded iteration is both a cost risk (§4) and a correctness risk (debate/self-refine loops can drift or converge on a shared wrong answer without an exit condition, §3 item 4). Don't remove this when adding new patterns.

6. **Reserve orchestrator/supervisor patterns (purrfect-blocks style) for tasks with genuine multi-domain parallelism; default to strict sequential pipelines (ceiling-designer style) otherwise.** The evidence is unambiguous that forcing coordination onto sequential work costs quality and money simultaneously. When building new `agent-smith` examples, pick the pattern based on whether the *task graph* has independent parallel branches — not based on which pattern is more novel or interesting to demonstrate.

7. **If adding a debate or voting mechanism as a new example pattern, scope it narrowly and pair it with a hard grounding step.** E.g., two independent reviewer instances voting on "does this meet acceptance criterion X" is fine and cheap; open-ended unbounded debate about subjective design quality is exactly the failure mode the drift/confident-convergence papers warn about (§3 item 4). If disagreement occurs, escalate to a human or a tool-grounded check — don't let the agents debate to resolution unsupervised.

8. **Instrument for the MAST failure categories explicitly** when building new agent-smith examples: log (a) whether task boundaries were ambiguous, (b) whether any handoff dropped context, (c) whether termination conditions were actually met vs. assumed. This maps well onto what ceiling-designer's Review Runs Log already partially captures and is worth generalizing across new examples.

9. **For genuinely parallelizable research/comparison work (Researcher stage, architecture-researcher), fan out with a cheap model and synthesize with a stronger one** — this is Anthropic's own production architecture (Opus lead, Sonnet workers) and is the most battle-tested version of "tiering pays off" in this research.

10. **Treat "more agents" as a latency/parallelism lever, not a quality lever, in any explanatory material (docs, this file, future READMEs) for `agent-smith`.** The strongest 2025-2026 evidence ([Tran & Kiela](https://arxiv.org/abs/2604.02460); Anthropic's own 80%-of-variance-is-token-spend finding) says the quality gains people attribute to "multi-agent" are usually actually gains from (a) more total compute/tokens or (b) better verification structure — both of which are achievable, and often cheaper, without adding more agents.

- [How to Train an LLM to do proofs: Beyond Verifiable Rewards](https://tobysimonds.com/research/2025/09/29/Proofs.html) (2025) — source of the ~15% generation vs. ~85% verification accuracy gap figure.
- [Trust but Verify! A Survey on Verification Design for Test-time Scaling](https://arxiv.org/pdf/2508.16665) (2025).

**Self-preference / judge bias**
- [Self-Preference Bias in LLM-as-a-Judge](https://arxiv.org/html/2410.21819v1) (2024) — GPT-4 self-preference bias -38% to +90% depending on benchmark.
- [Quantifying and Mitigating Self-Preference Bias of LLM Judges](https://arxiv.org/pdf/2604.22891) (2026).
- [Self-Preference Bias in Rubric-Based Evaluation of Large Language Models](https://arxiv.org/abs/2604.06996) (2026) — rubric decomposition cuts bias ~31.5%.

**Reflexion / Self-Refine**
- [Shinn, Cassano, et al. — "Reflexion: Language Agents with Verbal Reinforcement Learning"](https://arxiv.org/html/2303.11366) (2023). +22% absolute on AlfWorld across 12 iterative steps. 2023 — note the age; treat as foundational rather than current SOTA.

**Mixture of Agents**
- ["MoA is All You Need: Building LLM Research Team using Mixture of Agents"](https://arxiv.org/pdf/2409.07487) (2024).
- [Rethinking Mixture-of-Agents: Is Mixing Different LLMs Beneficial?](https://arxiv.org/abs/2502.00674) (2025) — proposes Self-MoA (single top model, resampled) beats mixed-model MoA by 6.6% on AlpacaEval 2.0; a useful caution against assuming model diversity itself helps.

**Best-of-N / process reward models**
- [TrajSelector: Harnessing Latent Representations for Efficient and Effective Best-of-N](https://arxiv.org/pdf/2510.16449) (2025) — 0.6B-parameter verifier sufficient for effective BoN reranking.

**Frameworks**
- [AutoGen handoffs / GroupChat docs](https://microsoft.github.io/autogen/dev//user-guide/core-user-guide/design-patterns/handoffs.html) — note AutoGen is now in maintenance mode per Microsoft, superseded by Microsoft Agent Framework.
- [OpenAI Swarm (archived reference)](https://github.com/openai/swarm) and its successor, the production [OpenAI Agents SDK](https://openai.github.io/openai-agents-python/multi_agent/) — the agents-as-tools vs. handoffs distinction is directly useful vocabulary for describing purrfect-blocks' orchestrator pattern vs. a hypothetical peer-handoff pattern.
- [CrewAI vs LangGraph vs AutoGen comparison (2026)](https://pecollective.com/blog/ai-agent-frameworks-compared/) — LangGraph now leads GitHub stars/enterprise adoption; role-based (CrewAI) vs. graph-based (LangGraph) vs. conversational (AutoGen) framing.
- [MetaGPT](https://ar5iv.labs.arxiv.org/html/2308.00352) (2023) and [ChatDev](https://arxiv.org/html/2307.07924v5) (2023) — early "AI software company" multi-agent architectures (CEO/PM/programmer/tester roles via chat chain). Both 2023 — dated relative to current orchestrator-worker practice, but historically important as the origin of role-based agent-team framing that CrewAI and purrfect-blocks both descend from. ChatDev reported outquality vs. MetaGPT on one metric (0.3953 vs 0.1523) while MetaGPT led on another (3.9 vs 2.1) — the metrics aren't directly comparable across the two papers' own eval setups, so treat both numbers as illustrative rather than a clean head-to-head.

**Cost/pricing**

- [BAMAS: Structuring Budget-Aware Multi-Agent Systems](https://arxiv.org/pdf/2511.21572) (2025) — data point on single-agent (121K tokens) vs multi-agent (63K tokens) token counts in one measured setting, illustrating that the token-cost direction is not universal and depends heavily on task structure.
